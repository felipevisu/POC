  • Harder to Debug & Search
    Since BEM class names can be long and very specific (like block__element--modifier), searching your codebase or browser dev tools for partial matches or related styles is tougher. You often have to type the entire class name exactly, which slows down debugging and style overrides.
  • Verbose Class Names
    BEM names can get long and harder to write manually.
  • Manual Enforcement
    Without tools or conventions, teams can be inconsistent with BEM.

Inline CSS

<div
  style="
    background: #ffeaea;
    border: 1px solid #e74c3c;
    color: #c0392b;
    padding: 1rem 1.5rem;
    border-radius: 0.5rem;
    margin: 1rem 0;
    max-width: 400px;
    box-shadow: 0 2px 8px rgba(0, 0, 0, 0.08);
    display: flex;
    flex-direction: column;
    gap: 0.5rem;
  "
>
  <div
    style="font-weight: bold; font-size: 1.1rem; margin-bottom: 0.25rem"
  >
    Error
  </div>
  <div style="font-size: 1rem; line-height: 1.4">
    Something went wrong. Please try again.
  </div>
</div>
  • Poor maintainability: Styles are scattered inside HTML elements, making it harder to update or manage across many elements.
  • No reusability: You can’t reuse inline styles easily; you have to repeat the same styles everywhere, causing duplication.
  • Harder to override: Inline styles have very high specificity, which can make overriding them with external CSS tricky.
  • Lack of separation of concerns: Mixing content (HTML) with presentation (CSS) violates best practices and reduces code clarity.
  • No media queries or pseudo-classes: Inline styles cannot handle media queries (@media) or pseudo-classes like :hover, limiting responsiveness and interactivity.
  • Larger HTML files: Inline styles increase the size of HTML files, potentially slowing down page load.
  • Performance issues: Excessive inline styling can cause the browser to recalculate styles more often, affecting performance.

  • Hard to Understand Component Purpose
    Utility classes like background-red border-red text-red describe styling but don’t convey semantic meaning, so just by looking at the class list it’s not obvious this is an alert component.
  • Class Name Clutter
    Elements end up with long lists of utility classes, which can be visually noisy and harder to read or maintain.
  • Poor Readability for Newcomers
    Developers unfamiliar with the utility classes have to learn the class names first before understanding the UI structure or styles.
  • Repetitive Markup
    Since styles are applied directly on elements via many utility classes, the same styles might be repeated across multiple elements, which could lead to duplication.

With the help of some tools like Purge.css integrated with my Vite or Webpack I clear all unused CSS classes: 


npm install @fullhuman/postcss-purgecss --save-dev

postcss.config.js


import purgecss from '@fullhuman/postcss-purgecss'

export default {
  plugins: [
    purgecss({
      content: ['./**/*.html', './src/**/*.{js,jsx,ts,tsx}'],
      defaultExtractor: content => content.match(/[\w-/:]+(?<!:)/g) || [],
    })
  ]
}

✅ Difference Between Built-in CSS (Build-time CSS) vs Runtime CSS

Aspect Build-time CSS (Built-in) Runtime CSS
⏱️ When it's applied During build process (e.g. npm run build) When the app runs in the browser
⚙️ How it's generated Pre-compiled into static CSS files Generated dynamically based on JS execution
📦 Bundle size Smaller (unused styles can be purged) Larger, often includes more styles than needed
🧠 Performance Fast – just static CSS applied Slower – CSS may be injected/manipulated in JS
📚 Example Libraries Tailwind (with purge), SCSS, PostCSS styled-components, Emotion, MUI sx, JSS
🧪 Dynamic features Harder (you need conditionals in class logic) Easier (you can use JS/props for styles)
